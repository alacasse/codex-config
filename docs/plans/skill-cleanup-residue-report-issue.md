# Clarify skill expectations for leftover cleanup residues

## Summary

Recent Graphify install-sandbox cleanup raised a reusable workflow question:
after a refactor completes, the skills are cleanup-biased, but they do not
clearly force a decision for small leftover test taxonomy or historical-evidence
markers that are no longer production code.

Concrete example from Graphify:
`tests/install_sandbox/test_spec_field_classification.py` introduced
`FIELD_CLASS_HISTORICAL_EVIDENCE` to classify
`historical_registry_yaml.platform_label` after internal synthetic label
vocabulary moved to `synthetic_result_label`.

The current skill guidance supports challenging this residue later, but there
is no concrete later batch or skill obligation that says this exact residue must
be removed once it stops carrying useful evidence.

## Triage

**Category:** workflow-design
**State:** ready-for-human-analysis

This is not a Graphify bug and not an immediate code deletion request. It is a
codex-config workflow question about whether repo-owned skills should require
explicit residue classification after refactors.

## Why This Matters

The current skills already say:

- do not preserve compatibility merely because it existed before;
- tests are evidence, not authority;
- migration-retention and topology-only tests are cleanup candidates; and
- final reports should document remaining compatibility paths.

That is directionally correct, but a small post-refactor residue can still land
in an ambiguous middle state:

- not production behavior;
- not public compatibility;
- not obviously dead on the same day it is introduced;
- useful as short-term guard evidence; and
- easy to forget if no removal condition or follow-up decision is recorded.

## Current Evidence

Relevant codex-config skill guidance:

- `skills/legacy-removal/SKILL.md` says speculative compatibility is a defect,
  compatibility needs a concrete reason, and obsolete internal-surface tests
  should be updated or deleted unless they protect a named external contract.
- `skills/dead-surface-audit/SKILL.md` says tests are not automatic liveness
  evidence and that test-only shape preservation is deletion-biased without a
  named contract.
- `skills/batch-runway/references/reporting-contracts-v1.md` says completion
  should not be treated as forecastable unless temporary compatibility logic is
  removed or intentionally documented.
- `skills/batch-runway/references/finalize-batch-v1.md` requires final reports
  to list remaining compatibility paths and deferred work.

Graphify case evidence:

- `FIELD_CLASS_HISTORICAL_EVIDENCE` is currently a test-only guard category,
  not production behavior.
- It was added to avoid re-promoting `platform_label` as current internal
  vocabulary.
- Local Graphify ledgers list broad deferred candidates such as
  legacy-reader deletion and docs reconciliation, but no concrete batch
  guarantees removal of this specific field class.

## Design Question

Should reusable skills require a post-refactor residue decision for leftover
test taxonomy, compatibility markers, historical-evidence buckets, aliases,
facades, or other temporary cleanup aids?

Possible outcomes:

1. Keep current guidance unchanged because the existing legacy-removal and
   dead-surface-audit skills are sufficient.
2. Add a Batch Runway closeout prompt/checklist item: classify all newly
   introduced compatibility or historical-evidence residues as `delete-now`,
   `keep-with-reason`, or `defer-with-removal-condition`.
3. Add legacy-removal guidance for short-lived test-only residues created during
   a cleanup batch: they may be valid only when they name the evidence they
   preserve and the condition under which they should be removed.
4. Add reviewer guidance so `runway_reviewer` flags residue without a concrete
   reason or follow-up condition.

## Proposed Direction

Analyze whether the reusable skills need a small, generic "residue closeout"
rule:

> When a refactor introduces or leaves behind test-only compatibility,
> historical-evidence, migration-retention, alias, facade, or old-vocabulary
> markers, closeout must classify each residue as removed, intentionally kept
> with a reason, or deferred with a removal condition.

This should stay project-neutral. Do not mention Graphify, install sandbox,
`FIELD_CLASS_HISTORICAL_EVIDENCE`, or `platform_label` in generic skill rules
except as optional local evidence in this report.

## Acceptance Criteria

- Review the current `legacy-removal`, `dead-surface-audit`, `batch-runway`
  reporting/finalization, and reviewer-routing guidance.
- Decide whether existing guidance is enough or whether a small generic
  residue-closeout rule is warranted.
- If guidance changes, update the relevant skill/reference files and
  `CHANGELOG.md`.
- Preserve reusable workflow ownership: no project-specific paths, validation
  commands, ledger names, or Graphify vocabulary should be embedded in generic
  skills.
- If no skill change is made, record why existing skills already cover the case
  and what a future agent should do when encountering similar residue.
